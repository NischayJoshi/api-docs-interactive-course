REST API Basics

Learn the fundamentals of REST APIs and how to document them effectively.

HTTP Methods

  • GET - Retrieve data from the server
  • POST - Create new resources
  • PUT - Update existing resources
  • PATCH - Partial updates to resources
  • DELETE - Remove resources

REST API Documentation Components

  • Endpoint URLs and paths
  • Request/Response formats
  • Authentication methods
  • Error codes and messages
  • Rate limiting information
  • Code examples in multiple languages

Example: GET Request Documentation

GET /api/users/{id}

Description: Retrieve a specific user by ID

Parameters:
  - id (path, required): The unique user identifier

Response 200 (application/json):
{
  "id": "123",
  "name": "John Doe",
  "email": "john@example.com"
}

OpenAPI Specification

OpenAPI (formerly Swagger) is a standard for describing REST APIs in a machine-readable format.

OpenAPI Benefits

  • Auto-generate interactive documentation
  • Create client SDKs automatically
  • API testing and validation
  • Standardized format across teams
  • Version control for API specifications

Example: OpenAPI 3.0 Specification

openapi: 3.0.0
info:
  title: User API
  version: 1.0.0
  description: API for managing users

paths:
  /users/{id}:
    get:
      summary: Get user by ID
      parameters:
        - name: id
          in: path
          required: true
          schema:
            type: string
      responses:
        '200':
          description: Successful response
          content:
            application/json:
              schema:
                type: object
                properties:
                  id:
                    type: string
                  name:
                    type: string
                  email:
                    type: string

GitBook Documentation

GitBook is a modern documentation platform that makes it easy to create beautiful, searchable docs.

GitBook Features

  • Markdown-based content authoring
  • Git integration for version control
  • Beautiful, responsive themes
  • Full-text search functionality
  • Custom domain support
  • Team collaboration tools

Setting Up GitBook

  1. Create a new GitBook space
  2. Connect your GitHub repository
  3. Organize content with SUMMARY.md
  4. Write content in Markdown files
  5. Publish and share your documentation

Example: SUMMARY.md Structure

# Summary

* [Introduction](README.md)
* [Getting Started](getting-started.md)
  * [Installation](getting-started/installation.md)
  * [Quick Start](getting-started/quick-start.md)
* [API Reference](api-reference.md)
  * [Authentication](api-reference/authentication.md)
  * [Endpoints](api-reference/endpoints.md)
* [Examples](examples.md)

Best Practices for API Documentation

Learn industry best practices for creating excellent API documentation.

Writing Guidelines

  • Be Clear and Concise: Use simple language and avoid jargon
  • Provide Examples: Include real-world code samples
  • Keep It Updated: Maintain documentation alongside code changes
  • Show Error Cases: Document error responses and edge cases
  • Include Getting Started: Help users quickly get up and running
  • Use Consistent Formatting: Maintain a uniform style throughout

Essential Documentation Sections

  • Overview and Introduction
  • Authentication Guide
  • API Reference (endpoints, parameters, responses)
  • Code Examples (multiple languages)
  • Error Handling
  • Rate Limiting and Quotas
  • Changelog and Versioning
  • FAQ and Troubleshooting

Practice Exercises

Exercise 1: Document a Simple Endpoint

Create documentation for a GET endpoint that retrieves a list of products. Include:

  • Endpoint URL
  • Query parameters (optional: category, limit, offset)
  • Response format with example
  • Possible error codes

Exercise 2: Write an OpenAPI Spec

Create an OpenAPI 3.0 specification for a blog API with:

  • Endpoints for posts (CRUD operations)
  • Authentication using API keys
  • Schema definitions for Post object

Exercise 3: GitBook Structure

Design a GitBook documentation structure for an e-commerce API including:

  • Navigation hierarchy
  • Table of contents (SUMMARY.md)
  • At least 5 main sections
